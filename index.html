Http

Posted on 2018-10-24

概述

HTTP是一种能够获取如HTML这样的网络资源的通信协议。它是在广域网上进行数据交换的基础，是一种clinet-server协议，也就是说，请求通常是由像浏览器这样的接收方发起的。一个完整的Web文档通常是由不同的子文档拼接而成的，像是文本、样式、图片、视频、脚本等。我们可以在chrome的调试工具的Network项看到这些分类，当然了，Network项中还包含了数据流，如XHR（XMLHttpReqest）、WS（Websocket）。

需要特别关注这样一个流程，浏览器B发送请求给服务器S，服务器S再返回数据给浏览器B。请求被叫做request，返回数据被叫做response，这两个名词在几乎所有的网络相关的API及RFC中都有出现，不论是chrome调试工具，还是nodejs对于http的API，抑或者是python抓包用到的requests模块。

浏览器与服务器并不一定是直接连接的——通常情况下都不是直接连接的。在他们之间，还有路由器、调制解调器等许多计算机。由于Web的层次设计，那些在网络层和传输层的细节都被隐藏起来了。HTTP位于最上层的应用层。虽然底层对于分析网络问题非常重要，但是大多跟对HTTP的描述不相干。大致的，HTTP传输过程中能够感知到的有三个部分：客户端（user-agent）、Web服务器、代理（Proxies）。客户端和Web服务器就不赘述了，但是代理还是需要在这里多说一些。在手机上调试前端代码是比较艰苦的，诚然我们可以用chrome调试工具的“仿造手机平板”功能，但有些只在手机上出现的问题chrome就没什么用了。这个时候怎么办呢？可以试试charles（Mac）或者fiddler（Windows），在手机wifi配置下代理，填入电脑的ip和软件监听的端口，手机通过电脑来访问服务器，每个http请求都可以在软件中看到。如果是加密了的https请求，也可以通过软件导入证书来解析。这样说起来，我也能理解为什么2012年左右学的Wireshark在前端不怎么火了，操作麻烦不说，还不一定有导入证书的功能。

请求报文（request）由这些元素组成：
一个HTTP的method，通常是由一个动词像GET POST 或者一个名词像OPTIONS，HEAD来定义客户端的动作行为。兴起的RESTful架构风格接口就要用到好几种method
要获取的资源的路径，通常是上下文中就很明显的元素资源的URL，不含protocol、domain、port
HTTP协议版本号。到目前为止最常用的应该还是HTTP/1.1，最新的是HTTP/2。
为服务端表达其他信息的可选头部headers。
对于一些像POST这样的方法，报文的body就包含了发送的资源（GET方法没有）

响应报文包含了下面的元素：
HTTP协议版本号
一个状态码。从1到5开头的三位数，具体含义很多地方都有，就不说了。
一个状态信息，这个信息是非权威的状态码描述信息，可以由服务端自行设定。RESTful架构风格接口其实也会用到响应报文的状态码和状态信息。不过以前我没接触过，更多的是在返回体中返回一个对象，再带code和msg字段。
HTTP headers，与请求头部类似
可选项，比起请求报文，响应报文中更常见地包含获取的资源body。当然啦，也有不含body的响应报文。试试请求method为HEAD。
